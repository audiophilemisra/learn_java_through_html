Functional Interfaces & The SAM Rule

A functional interface has exactly one abstract method (SAM — Single Abstract Method). The @FunctionalInterface annotation is optional but declares intent. 

@FunctionalInterface
public interface StringProcessor {
    String process(String input);
}

// Day 1 anonymous class:
StringProcessor upper = new StringProcessor() {
    @Override
    public String process(String input) {
        return input.toUpperCase();
    }
};

// Day 2 lambda — same interface, 80% less code:
StringProcessor upper = input -> input.toUpperCase();
Lambda Syntax Variants
// Parameter inference (most common)
(String s) -> s.length()

// Type inferred (shorter)
s -> s.length()

// Multi-line body with curly braces
(x, y) -> {
    double result = x + y;
    return result;
}

// No parameters
() -> System.out.println("Hello")

Lambdas can capture local variables, but the variable must be effectively final — never reassigned after initialization. This is the closure trap.

The #1 Pain Point: Checked Exceptions Inside Lambdas

You cannot throw IOException from Stream.map() — functional interfaces don't declare checked exceptions.

Survival Pattern 1: Do I/O before the stream.

// Read everything first, then stream the in-memory list
List<String> lines = Files.readAllLines(path);  // IOException handled here
lines.stream()
     .filter(line -> !line.isBlank())
     .forEach(System.out::println);

Survival Pattern 2: Wrapper functions.

// Convert checked to unchecked
public static <T> Supplier<T> unchecked(CheckedSupplier<T> supplier) {
    return () -> {
        try { return supplier.get(); }
        catch (Exception e) { throw new RuntimeException(e); }
    };
}

Terminal Operations — reduce() & collect()
Reduce and collect stream results. Never mutate shared state.
forEach vs. forEachOrdered
// forEach — no order guarantee (especially in parallel)
list.parallelStream().forEach(System.out::println);

// forEachOrdered — preserves encounter order

Never Mutate Shared State Inside forEach()
// WRONG — race condition in parallel
ArrayList<Integer> results = new ArrayList<>();
list.parallelStream().forEach(results::add); // UNSAFE!

// CORRECT — use collect()
List<Integer> results = list.parallelStream()
    .collect(Collectors.toList());
reduce() — Immutable Reduction
// Identity, accumulator, combiner
int sum = numbers.stream()
    .reduce(
        0,                           // Identity: starting value
        (int a, int b) -> a + b,     // Accumulator: combine two elements
        Integer::sum              // Combiner: merge partial results (parallel)
    );

// Associativity requirement: (a op b) op c == a op (b op c)
// The combiner is only used in parallel streams, but always provide it.

Hour 10
Synchronization & volatile
Fix race conditions. Understand visibility vs. atomicity.
synchronized — Intrinsic Lock (Monitor)
// Method-level synchronization
public synchronized void increment() {
    count++;  // Only one thread at a time
}

// Block-level synchronization (finer-grained)
public void increment() {
    synchronized (this) {  // Lock on this object
        count++;
    }
}

Lock granularity: Synchronize on the smallest scope that protects the data. Method-level is simpler but creates a bottleneck if only one field needs protection.

volatile — Visibility Guarantee
private volatile boolean running = true;

public void stop() {
    running = false;  // Visible to all threads immediately
}

public void run() {
    while (running) {
        // Do work
    }
    System.out.println("Stopped cleanly");
}
volatile Does NOT Make Increment Atomic

volatile guarantees visibility — writes are flushed to main memory. It does NOT guarantee atomicity. count++ is still read-modify-write (three operations). For atomic increment, use AtomicInteger (Hour 15).

Hour 11
Java Memory Model & Thread Interruption
Understand happens-before. Master the interruption contract.
Java Memory Model — Happens-Before
  • Atomicity: Which operations cannot be interleaved?
  • Visibility: When does one thread see another's writes?
  • Ordering: Can the compiler/CPU reorder statements?

Happens-before guarantees: a write to a volatile field happens-before every subsequent read of that field. A synchronized block exit happens-before the entry of the next synchronized block on the same monitor.

Why volatile Is Not Enough for Increment

count++ is read-modify-write: three operations. volatile ensures the read sees the latest write, but another thread can write between the read and the write. You need synchronized or AtomicInteger.

Thread Interruption Contract (Critical)
// The canonical worker loop pattern
while (!Thread.currentThread().isInterrupted()) {
    try {
        // Do work
        Thread.sleep(100);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();  // RESTORE THE FLAG!
        break;  // Exit gracefully
    }
}
Why Catching InterruptedException Must Restore the Flag

When InterruptedException is thrown, the thread's interrupt status is cleared. If you swallow the exception without restoring the flag, higher-level code cannot detect the interruption. Always call Thread.currentThread().interrupt() in the catch block before breaking.

ThreadLocal — Per-Thread Storage
private static final ThreadLocal<String> requestId =
    ThreadLocal.withInitial(() -> "REQ-" + ThreadLocalRandom.current().nextInt(1000));

// Each thread gets its own copy
System.out.println(requestId.get());  // Thread-safe, no synchronization needed

Hour 12
Inter-Thread Communication & Classic Problems
Low-level coordination. Understand deadlock.
wait() / notify() / notifyAll()
  • Must own the monitor — call inside synchronized block on the same object.
  • Always use while loops — not if — to guard against spurious wakeups.
  • notify() wakes one waiting thread; notifyAll() wakes all. Prefer notifyAll() unless you are sure only one thread should proceed.
synchronized (lock) {
    while (!condition) {   // ALWAYS while, never if
        lock.wait();         // Releases lock, waits for notify
    }
    // Condition is now true — proceed
}
Producer-Consumer with wait/notify
class BoundedBuffer<T> {
    private final Queue<T> queue = new LinkedList<>();
    private final int capacity;

    public synchronized void put(T item) throws InterruptedException {
        while (queue.size() == capacity) wait();
        queue.add(item);
        notifyAll();
    }

    public synchronized T take() throws InterruptedException {
        while (queue.isEmpty()) wait();
        T item = queue.poll();
        notifyAll();
        return item;
    }
}
Deadlock
// Thread 1 locks A then B. Thread 2 locks B then A.
// Both wait forever. Prevention: always lock in the same order.
synchronized (lockA) {     // Thread 1 holds A
    synchronized (lockB) { // Waits for B (held by Thread 2)
        // Deadlock!
    }
}

// Detection: thread dump (jstack, VisualVM, IntelliJ debugger).

Livelock: Threads keep reacting to each other but make no progress. Starvation: A thread never gets CPU time due to greedy peers. Both are brief exposure topics.

Hour 13
java.util.concurrent — Executors
Replace raw threads with managed thread pools.
Why Raw Threads Are Bad
  • Expensive: Each thread consumes ~1MB stack memory and OS resources.
  • No reuse: Threads are created and destroyed for each task.
  • Unbounded: Creating too many threads crashes the JVM with OutOfMemoryError.

Solution: Thread pools — reuse a fixed set of threads for many tasks.

Executor Framework Hierarchy
Executor               // void execute(Runnable)
  └── ExecutorService      // submit(), shutdown(), invokeAll()
        └── ScheduledExecutorService // schedule(), scheduleAtFixedRate()
Thread Pool Types
Factory MethodBehaviorWhen to Use
newFixedThreadPool(n)n threads, fixed. Tasks queue if all busy.General purpose. Most common choice.
newCachedThreadPool()Creates threads on demand, reuses idle threads.Short-lived, async tasks. Beware unbounded growth.
newSingleThreadExecutor()One thread. Tasks executed sequentially.When order must be preserved.
newScheduledThreadPool(n)n threads for delayed/periodic tasks.Cron-like scheduling, retry delays.
Submitting Tasks & Getting Results
ExecutorService pool = Executors.newFixedThreadPool(4);

// execute — fire and forget (Runnable)
pool.execute(() -> System.out.println("Task ran"));

// submit — returns Future<T> (Callable)
Future<Integer> future = pool.submit(() -> {
    return 42;  // Callable returns a value
});
System.out.println(future.get());  // Blocks until result available
Runnable vs. Callable — Recall from Hour 9

execute(Runnable) returns void. submit(Callable<T>) returns Future<T> — the same Runnable vs. Callable distinction from Hour 9.

Shutting Down Properly
pool.shutdown();  // No new tasks. Finish submitted tasks.

// If tasks won't finish:
if (!pool.awaitTermination(60, TimeUnit.SECONDS)) {
    pool.shutdownNow();  // Force interrupt (Hour 11 pattern)
}

Concurrent Collections & Blocking Queues
Use thread-safe collections. Build better producer-consumer.
ConcurrentHashMap
ConcurrentHashMap<String, Integer> map = new ConcurrentHashMap<>();

// Atomic compute — no race conditions
map.computeIfAbsent("key", k -> 0);
map.merge("key", 1, Integer::sum);  // If key exists, sum; if not, set 1

// Thread-safe iteration
map.forEach(1000, (k, v) -> System.out.println(k + "=" + v));

Uses lock striping / CAS — multiple threads can read and write concurrently without locking the entire map. Beats Collections.synchronizedMap() which locks the whole map on every operation.

CopyOnWriteArrayList / CopyOnWriteArraySet
Use Only When Reads Vastly Outnumber Writes

Every write copies the entire array — expensive but lock-free for readers. Ideal for event listener lists or configuration that rarely changes.

    for (int i = 0; i < 20; i++) System.out.println(queue.take());
})).start();

Code shrinks ~80%. No synchronized, no wait, no notify — the queue handles everything.

Hour 15
Locks, Atomics & Synchronization Utilities
Advanced concurrency primitives. From locks to latches.
ReentrantLock — Explicit Locking
ReentrantLock lock = new ReentrantLock();

lock.lock();
try {
    // Critical section
} finally {
    lock.unlock();  // ALWAYS in finally
}

// Benefits over synchronized:
// tryLock()        — non-blocking attempt
// tryLock(timeout) — timed attempt
// lockInterruptibly() — interruptible
// newCondition()   — Condition variables (like wait/notify but better)
ReadWriteLock — Many Readers, One Writer
ReadWriteLock rwLock = new ReentrantReadWriteLock();

// Multiple threads can read simultaneously
rwLock.readLock().lock();
try { // read data } finally { rwLock.readLock().unlock(); }

// Only one thread can write; blocks all readers
rwLock.writeLock().lock();
try { // write data } finally { rwLock.writeLock().unlock(); }
Atomic Classes & CAS
AtomicInteger counter = new AtomicInteger(0);
counter.incrementAndGet();    // Atomic ++
counter.compareAndSet(5, 10); // If current==5, set to 10

AtomicReference<String> ref = new AtomicReference<>("initial");
ref.compareAndSet("initial", "updated");  // CAS
CAS (Compare-And-Swap)

A hardware primitive that atomically compares a value and swaps it only if it matches the expected value. All Atomic* classes use CAS under the hood — no locks needed. The ABA problem (value changes A→B→A, CAS thinks nothing changed) exists but is rare in practice.

LongAdder & LongAccumulator

LongAdder is a high-contention optimization over AtomicLong. Each thread maintains its own counter; the sum is computed on read. 

LongAdder hitCounter = new LongAdder();

// Each thread increments independently — no CAS contention
hitCounter.increment();

// Read the sum (adds all thread-local counters)
long total = hitCounter.sum();

CountDownLatch latch = new CountDownLatch(4);

Hour 16
CompletableFuture — Async Java
Compose asynchronous operations without blocking.
Why CompletableFuture? The Future Limitation
Future<Integer> future = pool.submit(() -> compute());
int result = future.get();  // BLOCKS until done. Cannot chain.

Future.get() is blocking. CompletableFuture lets you chain, combine, and compose async operations without ever calling get() until you need the result.

Creation
// Run async, return a value
CompletableFuture<String> cf = CompletableFuture.supplyAsync(() -> {
    return fetchUserProfile();  // Runs in ForkJoinPool.commonPool()
});

// Run async, no return value
CompletableFuture<Void> cf = CompletableFuture.runAsync(() -> {
    sendEmail();
});
Custom Executor for I/O Tasks

supplyAsync() uses ForkJoinPool.commonPool() by default — the same pool used by parallel streams. Pass a custom Executor for blocking I/O tasks so you don't starve CPU-bound work.

Transformation
// thenApply — transform the result (T → R)
cf.thenApply(profile -> profile.getName())

// thenAccept — consume the result (T → void)
cf.thenAccept(name -> System.out.println(name))

// thenRun — execute after completion (no input)
cf.thenRun(() -> System.out.println("Done"))
Composition & Combination
// thenCompose — dependent async call (flatMap for futures)
CompletableFuture<List<Order>> orders = fetchUser(userId)
    .thenCompose(user -> fetchOrders(user.getId()));

// thenCombine — independent async calls, merge results
CompletableFuture<Double> total = fetchPrice()
    .thenCombine(fetchTax(), (price, tax) -> price + tax);

// allOf — wait for all futures
CompletableFuture.allOf(cf1, cf2, cf3).join();

// anyOf — proceed on first completion
CompletableFuture.anyOf(cf1, cf2, cf3).thenAccept(result -> ...);
Error Handling
// exceptionally — handle specific error
cf.exceptionally(ex -> {
    System.err.println("Failed: " + ex.getMessage());
    return "default";  // Fallback value
});

// handle — both success and failure
cf.handle((result, ex) -> {
    if (ex != null) return "fallback";
    return result;
});

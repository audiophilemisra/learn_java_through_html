Why Resources Matter

Testing Concurrent Code
@Test
void concurrentIncrementDoesNotLoseData() throws InterruptedException {
    AtomicInteger counter = new AtomicInteger(0);
    int threadCount = 10;
    CountDownLatch startGate = new CountDownLatch(1);
    CountDownLatch endGate = new CountDownLatch(threadCount);

    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            try { startGate.await(); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
            counter.incrementAndGet();
            endGate.countDown();
        }).start();
    }

    startGate.countDown();  // All threads start simultaneously
    endGate.await();          // Wait for all to finish
    assertEquals(threadCount, counter.get());
}

Never Use Statement — Always Use PreparedStatement
// NEVER: vulnerable to SQL injection
Statement stmt = conn.createStatement();
stmt.executeQuery("SELECT * FROM users WHERE name = '" + userInput + "'");

// ALWAYS: parameterized, safe from injection
PreparedStatement stmt = conn.prepareStatement("SELECT * FROM users WHERE name = ?");
stmt.setString(1, userInput);

Let the Database Do the Work

Know what GROUP BY, COUNT(), SUM(), and AVG() can do. Don't re-implement in Java what SQL handles natively. Fetch aggregated results from the database rather than pulling all rows and computing in Java.

Transactions
try (Connection conn = DriverManager.getConnection(url)) {
    conn.setAutoCommit(false);  // Start transaction

    try {
        // Multiple operations as one unit
        insertOrder(conn, order);
        updateInventory(conn, order);
        conn.commit();           // All succeed → commit
    } catch (Exception e) {
        conn.rollback();          // Any failure → undo everything
        throw e;
    }
}
Without Explicit Transactions, Every Statement Auto-Commits

This is dangerous in concurrent code. If your application crashes between two operations (e.g., after inserting an order but before updating inventory), your data is inconsistent. Always group related operations inside a transaction block with setAutoCommit(false).

Text Blocks (Java 15+) — Cleaner SQL
String sql = """
    SELECT e.name, e.salary
    FROM employees e
    WHERE e.dept = ?
    """;

// Much cleaner than string concatenation:

Try-with-Resources Revisited

Connection, PreparedStatement, and ResultSet all implement AutoCloseable. Nest them in try-with-resources to guarantee cleanup. 

try (Connection conn = DriverManager.getConnection(url);
     PreparedStatement stmt = conn.prepareStatement(sql);
     ResultSet rs = stmt.executeQuery()) {
    // All three auto-closed when block exits
} catch (SQLException e) {
    throw new RuntimeException("Database error", e);
}

Hour 6
The DAO Pattern
Separate data access from business logic. Test your persistence.
Data Access Object — The Abstraction
public interface EmployeeDao {
    Employee save(Employee employee);
    Optional<Employee> findById(int id);
    List<Employee> findAll();
    void delete(int id);
}

The DAO pattern separates what data operations exist (interface) from how they're implemented (H2, PostgreSQL, etc.). Your business code depends on the interface, not the implementation.

H2EmployeeDao Implementation
public class H2EmployeeDao implements EmployeeDao {
    private final Connection connection;

    public H2EmployeeDao(Connection connection) {
        this.connection = connection;
    }

    @Override
    public Employee save(Employee employee) {
        String sql = """
            INSERT INTO employees (name, salary, dept)
            VALUES (?, ?, ?)
            """;
        try (PreparedStatement stmt = connection.prepareStatement(sql,
                Statement.RETURN_GENERATED_KEYS)) {
            stmt.setString(1, employee.name());
            stmt.setDouble(2, employee.salary());
            stmt.setString(3, employee.dept());
            stmt.executeUpdate();
            try (ResultSet rs = stmt.getGeneratedKeys()) {
                if (rs.next()) {
                    return new Employee(rs.getString("name"),
                        rs.getDouble("salary"), rs.getString("dept"));
                }
            }
        } catch (SQLException e) {
            throw new RuntimeException(e);
        }
        return employee;
    }

    // findById, findAll, delete follow the same pattern
}
DAO in Tests
@Test
void saveAndFindById() {
    EmployeeDao dao = new H2EmployeeDao(connection);  // H2 in-memory
    Employee saved = dao.save(new Employee("Alice", 85000, "Engineering"));
    Employee found = dao.findById(saved.id());
    assertEquals(saved, found);  // Round-trip verified
}
Connection Pooling (Conceptual)

In production, you would use a connection pool (e.g., HikariCP) instead of creating a new DriverManager.getConnection() per operation. Creating connections is expensive — pools maintain a set of reusable connections. For this course, H2's in-memory mode is sufficient, but know that connection pools are the production standard.

Hour 7
Logging with SLF4J + Logback
Replace System.out.println forever. Observe thread behavior in logs.
Why Logging Matters
  • Levels: Filter by severity — trace in development, warn in production.
  • Context: Timestamps, thread names, class names — everything System.out.println lacks.
  • Performance: Parameterized logging skips string concatenation when the level is disabled.
  • Destination: Console, file, remote — configurable without changing code.
Log Levels

Using SLF4J in Code
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class OrderProcessor {
    private static final Logger log = LoggerFactory.getLogger(OrderProcessor.class);

    public void process(Order order) {
        log.info("Processing order {}", order.id());  // Parameterized — no string concat
        log.debug("Order details: amount={}, product={}", order.amount(), order.product());

        try {
            persistOrder(order);
        } catch (SQLException e) {
            log.error("Failed to persist order {}", order.id(), e);  // Exception appended
        }
    }
}
logback.xml Configuration
<!-- src/main/resources/logback.xml -->
<configuration>
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>%d{HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <appender name="FILE" class="ch.qos.logback.core.FileAppender">
        <file>logs/application.log</file>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss} [%thread] %-5level %logger - %msg%n</pattern>
        </encoder>
    </appender>

    <root level="INFO">
        <appender-ref ref="CONSOLE" />
        <appender-ref ref="FILE" />
    </root>
    <logger name="com.example" level="DEBUG" />
</configuration>
Parameterized Logging Avoids String Concat Overhead

logger.info("Processing order {}") — the {} is only replaced if INFO level is enabled. No string concatenation cost when the level is disabled. Never use logger.info("Processing " + order.getId()) — that concatenation runs regardless of log level.

4 Persist Day 2 H13 Day 3 H5-H6

Use ExecutorService with a fixed thread pool. Each worker inserts valid orders into H2 via your DAO pattern. Wrap inserts in a transaction:

ExecutorService pool = Executors.newFixedThreadPool(4);

for (Order order : validOrders) {
    pool.submit(() -> {
        try (Connection conn = getConnection()) {
            conn.setAutoCommit(false);
            try {
                dao.save(conn, order);
                conn.commit();
            } catch (Exception e) {
                conn.rollback();
                log.error("Failed to persist order {}", order.id(), e);
            }
        }
    });
}
5 Concurrent Safety Day 2 H15

Use CountDownLatch to ensure all orders are persisted before generating the report:

CountDownLatch latch = new CountDownLatch(validOrders.size());

for (Order order : validOrders) {
    pool.submit(() -> {
        try {
            persistOrder(order);
        } finally {
            latch.countDown();
        }
    });
}

latch.await();  // All inserts complete before report
